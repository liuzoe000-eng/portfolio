# 作品集项目页内容提炼提示词

> 用途：快速将一个新项目的原始素材，提炼成符合本作品集风格的完整页面内容。
> 使用方式：把这个提示词发给 AI，然后粘贴你的项目原始资料（背景、方案、数据、复盘等），AI 会按格式输出可直接填入 HTML 的内容。

---

## 提示词正文

```
你是一位擅长 UX 作品集写作的设计顾问，帮我把以下项目原始资料，提炼成一份结构化的作品集页面内容。

【输出要求】

请严格按照以下 7 个模块输出，每个模块的内容要求如下：

---

### 模块 1：首页卡片（index.html 用）

输出 3 个字段：
- work-hook（一句话钩子）：用一句话说清楚"这个项目解决了什么本质问题"，不超过 25 字，要有张力，不能是平铺直叙的功能描述。格式：问题现象 + 设计切入点，或 反常识结论 + 行动。
- work-sub（副标题）：格式固定为「核心数据 · 角色 · 业务线」，例如「屏效提升 ~10% · 主R · 外卖搜索」
- work-tags（2个标签）：提炼这个项目最核心的两个方法论标签，不是功能描述，是设计能力标签，例如「系统化重构」「A/B 验证」「用研驱动」「全链路视角」

---

### 模块 2：项目 Hero 区

输出 2 个字段：
- 项目大标题（h1）：格式「动词 + 核心矛盾/转变」，要有叙事感，例如「从堆砌到系统——一次让屏效提升 10% 的框架重构」
- 项目一句话定位（tagline）：用一句话说清楚这个项目"为什么值得被看"，强调设计的独特视角或反常识判断，不超过 30 字

---

### 模块 3：数据指标条（metrics bar）

输出 3～4 个核心数据，每个数据包含：
- 数值（要醒目，带单位，例如 +0.31pp / ~+10% / +63%）
- 标签（说明这个数据是什么，不超过 10 字）

选数据的原则：优先选有具体数字的结果指标；没有数字的用行为指标或覆盖规模替代；不要用"↑"这种模糊表达。

---

### 模块 4：正文章节（section）

按项目的叙事逻辑，输出 4～6 个章节，每个章节包含：
- section-num（章节编号 + 主题词）：格式「01 — 问题定位」，主题词 4～6 字，点明这一节在讲什么
- h2（章节标题）：要有观点，不是描述性的，要让读者产生"为什么"的好奇，例如「最难的决策：用数据否定了「显而易见」的方案」
- 正文段落：2～4 段，每段 60～120 字，语气是"设计师在复盘自己的判断"，不是汇报，不是说明书。要体现：发现了什么问题 → 做了什么判断 → 为什么这么判断 → 结果怎样

章节叙事顺序建议（可根据项目调整）：
1. 问题定位：这个项目的核心矛盾是什么，为什么到了必须改的临界点
2. 设计北极星 / 框架定义：用什么视角/框架来拆解这个问题
3. 核心设计决策（可拆成 2 个章节）：最关键的 1～2 个设计动作，重点写判断过程，不是方案罗列
4. 项目成果：数据结果 + 业务影响
5. 设计者视角（复盘）：见下方模块 5

**双轨并行项目的章节拆法（如同时有"心智建设"和"体验优化"两条线）：**
- 章节3 = 目标1的完整叙事：先放策略框架图，再按策略编号顺序（策略1→策略2）放落地图
- 章节4 = 目标2的完整叙事：同样先策略框架，再具体页面/Before-After
- 两个目标之间不要交叉穿插，每个目标内部保持"策略→落地"的完整闭环
- 如有 AI 能力/技术提效等横向能力，单独作为一个章节放在两个目标之后

---

### 模块 5：复盘卡片（card-num + kw 格式）

输出 4 张复盘卡片，每张包含：
- card-num：01 / 02 / 03 / 04
- 卡片标题（h4）：一个值得被记住的判断或教训，6～12 字，有观点
- 卡片正文：80～120 字，讲清楚"这个判断为什么难、为什么对"，语气是设计师的第一人称反思
- kw 关键词：在正文中标注 2～4 个关键词（用 <span class="kw"> 包裹），选那些"如果 hover 高亮，读者会觉得这是核心信息"的词，通常是方法论词汇、反常识结论、或关键数据

4 张卡片的内容方向建议：
1. 定义/框架类：这个项目最重要的概念定义或框架搭建
2. 优先级/取舍类：面对多个方向，如何做取舍，依据是什么
3. 协同/推动类：设计如何影响产品/业务方向，或跨团队推进的关键动作
4. 验证/度量类：如何设计验证方案，如何判断结果是真实改善还是噪音

---

### 模块 6：首页 detail overlay 数据（index.html 的 JS 数据对象）

输出一个 JS 对象，格式如下：
{
  title: '项目大标题',
  cover: '封面图文件名（如 cover-xxx.webp，没有则填 null）',
  coverBg: '渐变背景色（格式：linear-gradient(135deg, #颜色1, #颜色2)，颜色要与项目气质匹配）',
  tags: ['标签1', '标签2', '标签3'],
  accentColor: '#强调色（与 coverBg 呼应）',
  sections: [
    { label: '项目背景', text: '150字以内，说清楚背景和核心矛盾' },
    { label: '设计策略', text: '150字以内，说清楚核心设计动作和判断逻辑' },
    { label: '我的角色', text: '50字以内，说清楚参与度和负责模块' },
  ],
  results: [
    { val: '数值', lbl: '指标名称' },
    ...
  ]
}

---

### 模块 7：内容质检清单

输出完以上内容后，自我检查以下 8 项，逐条给出 ✅ 或 ❌ + 说明：

1. work-hook 是否有张力，避免了"我们做了XXX功能"的平铺直叙？
2. 每个章节标题是否有观点，而不是描述性的？
3. 复盘卡片的 kw 关键词是否选了真正的核心信息，而不是随机加粗？
4. 数据指标是否都有具体数字，避免了模糊的"↑"？
5. 正文语气是否是"设计师在复盘判断"，而不是"项目汇报"或"功能说明"？
6. 4 张复盘卡片是否覆盖了定义/取舍/协同/验证四个维度？
7. 每张图片是否已读清楚内容，并归属到正确章节？（不能凭文件名猜测，要打开图片确认）
8. 同一章节内的图片顺序是否遵循"策略框架先于具体页面"、"策略1先于策略2"的原则？

---

【我的项目原始资料】

（在这里粘贴你的项目背景、方案描述、数据结果、复盘笔记等原始内容，格式不限）
```

---

## 使用说明

**第一步：准备原始素材**

把以下内容整理好，粘贴到提示词末尾的「原始资料」区域：
- 项目背景（为什么做这个项目，当时的业务问题是什么）
- 核心设计方案（做了什么，重点是判断过程，不是方案列表）
- 数据结果（越具体越好，有 A/B 数据最佳）
- 你觉得这个项目最值得说的 1～2 个点（可以是方法论、反常识判断、或跨团队推动）

**第二步：运行提示词，拿到输出**

把提示词 + 原始素材发给 AI，拿到 7 个模块的输出。

**第三步：对照本作品集的 HTML 结构填入**

参考 project-02.html 的结构（最完整的参考案例），把内容填入对应位置：
- 模块 1 → index.html 的 `.work-card` 区域
- 模块 2 → project-xx.html 的 `.hero-section` 区域
- 模块 3 → project-xx.html 的 `.metrics-bar` 区域
- 模块 4 → project-xx.html 的各 `.section` 区域
- 模块 5 → project-xx.html 的复盘章节 `.two-col` + `.card` 区域
- 模块 6 → index.html 的 `PROJECT_DATA` JS 对象
- 模块 7 → 自检，不满足的项目回去修改

**第四步：配图（先建对照表，再填入 HTML）**

图片命名规则：`s{项目编号}-{内容描述}.webp`，例如 `s4-strategy.webp`。
图片放在 portfolio-static 目录下，在 HTML 中用 `<img src="s4-xxx.webp">` 引用。

**填图前必须先做图片归属对照表**，格式如下：

| 图片文件名 | 图片实际内容（读图后填写） | 归属章节 | 章节内排序 |
|---|---|---|---|
| s4-strategy.webp | 设计难点&策略框架 | 01 问题定位 | 1 |
| s4-fullchain.webp | 全链路方案总览 | 02 设计北极星 | 1 |
| ... | ... | ... | ... |

**图片排序的两条铁律：**
1. **策略框架图永远先于具体页面图** — 先展示"为什么这么做"，再展示"做了什么"。路径提效策略图要在搜索结果页截图之前。
2. **同一目标内，按策略编号顺序排** — 如果图片内容标注了"策略1""策略2"，必须严格按编号顺序，不能因为视觉好看就调换。

**图片归属判断方法：**
- 先在浏览器里逐张打开图片，读清楚图片标题/标注的内容
- 对照复盘文档的章节结构，判断这张图属于哪个目标/策略
- 同一章节内有多张图时，按"总→分"顺序排列（先放总览/框架图，再放具体方案/Before-After）

## 本作品集的风格原则（写内容时始终对照）

**内容层面**
- 每个项目必须有一个"反常识判断"：数据推翻了直觉、用户行为出乎意料、或设计切入点与常规不同
- 复盘不是总结，是"如果重来一次，这个判断我还会这么做吗，为什么"
- 数据要有上下文：不只说"+10%"，要说"+10% 的是什么，在什么条件下"

**表达层面**
- 章节标题要有观点，不是描述。"优化商家卡片布局"是描述，"从信息堆砌到差异化说服"是观点
- kw 关键词是给快速扫读者准备的，hover 高亮时要让人觉得"这就是核心"
- 语气是设计师的第一人称，不是产品文档，不是汇报 PPT

**视觉层面**
- 每个项目有自己的 accent color，贯穿 cover gradient、card-num 背景、kw 下划线
- 配图优先用真实截图，其次是方案对比图（before/after），最后才是示意图
- 图片比例统一，section 内图片宽度撑满内容区
